The Only Testing Metric That Matters: Confidence

The topic of testing in software has always generated a lot of buzz. Some engineers swear by 100% coverage. Others treat every test as a maintenance liability. Most teams land somewhere in the middle, guided more by convention than by any coherent principle.

I try to take a very simple view on this. A good test does exactly one thing — it increases your confidence in the system. That’s it. That’s the whole metric.

Classification Is a Distraction

The endless categorization of tests — unit, integration, end-to-end, contract, smoke — is mostly noise when it comes to deciding what to write. When I’m working on a complex function and I’m not quite sure it behaves the way I think, I write a test. Maybe that test happens to be a unit test, but the type wasn’t the decision. The uncertainty was. The classification is just a label that falls out of what I needed to verify.

Methodologies that put coverage targets and test taxonomy at the center tend to produce suites that look impressive and feel hollow. Full coverage that still has you crossing your fingers at deployment time isn’t useful. A green CI badge that doesn’t move the needle on your actual confidence in the system is also not useful.

The question I come back to is simple: when the suite goes green, do I feel better about shipping? If the answer is yes, the tests are doing their job. If the answer is no, something is wrong — regardless of what the coverage report says.

Don’t Dismiss the Simple Test

“Test the places that make you uneasy” is personal advice, but I think it works in both directions. A senior engineer who’s lived with a system for years might have blind spots baked in — places where familiarity has quietly replaced vigilance. A junior engineer or newcomer might be looking at a real gap that everyone else stopped seeing.

Be careful about labeling a test “overly cautious.” What looks like an obvious or trivial assertion may be encoding a key assumption about how the system behaves. Consider something like this: 

it('does not apply discount to free items', () => {
  const result = applyDiscount({ price: 0, discountPercent: 10 });
  expect(result.price).toBe(0);
});

It looks obvious, and it is — in its current state at least. But this test encodes a business rule. The moment someone refactors the discount logic and accidentally breaks it, this is the thing that catches it. These simple, seemingly obvious tests are often the best protection against regression — and they’re exactly the kind that get skipped when experience tips into overconfidence.

That said, tests can also drift into pedantry. There’s a meaningful difference between testing behavior and testing incidental details. A good example: do you need to assert the exact wording of an error message? For most systems, no. The important thing is that the error is thrown. Asserting the specific string makes the test brittle — it will break any time someone tweaks the message for clarity, even if nothing about the underlying behavior changed. If the message text is genuinely business-critical, that’s a different conversation. But if it isn’t, the assertion is noise.

The Most Wasteful Test in the Wild

There’s one specific pattern I’ve seen repeatedly that I’d nominate as the most common form of wasted testing effort: mocking a dependency and then testing the mock.

It looks like this:

async function getUser(id: string): Promise<User> {
  return await apiClient.get(`/users/${id}`);
}

it('calls the API and returns a user', async () => {
  const mockUser = { id: '1', name: 'Alice' };
  jest.spyOn(apiClient, 'get').mockResolvedValue(mockUser);

  const result = await getUser('1');

  expect(apiClient.get).toHaveBeenCalledWith('/users/1');
  expect(result).toEqual(mockUser);
});

This test doesn’t verify any real behavior. It confirms that the mock returns what you told it to return. That’s not a test — it’s a script that reassures you the mocking library works. It adds nothing to your confidence, and it adds a lot to your maintenance burden. These tests are brittle. The frustration they generate is real and the value they provide is not.

Delete these tests. If a test doesn’t increase your confidence in the system, it is taking up space and manufacturing cognitive dissonance. Deleting it is not heresy — it’s honesty.

The Most Valuable Tests You Can Write

Counterintuitively, some of the most valuable tests you’ll ever write are the ones you write after something breaks unexpectedly.

When you hit an error you didn’t anticipate — something you didn’t fully think through, or simply didn’t know you didn’t know — that’s a test waiting to be written. Capture the failure as a failing test first. Then fix it by making the test pass. This approach does three things simultaneously: it resolves the bug, protects you from ever being surprised by that same failure again, and most importantly elevates your confidence in the system.

Regression prevention is among a test’s most important duties. Tests written in response to real, unexpected failures tend to be grounded in actual system behavior rather than assumptions about it. They earn their place.

A Simple Principle for a Trustworthy Suite

If I’m starting fresh, the guide is always the same: test what you’re unsure of, and test what has surprised you. Skip the ceremony. Skip the coverage games. Ask, honestly, whether the suite makes you more confident — and if the answer for any given test is no, let it go.

Use tests to make you and your team more confident. That’s what they’re for.
