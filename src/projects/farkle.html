Farkle

A real-time multiplayer implementation of the dice game Farkle. Players create or join a room via a shareable link, then take turns rolling six dice and selecting scoring combinations. On each turn you can keep rolling your remaining dice to press your luck, or bank your accumulated points to lock them in. Roll no scoring dice and you farkle — losing everything you built up that turn. First player to 10,000 points triggers a final round so every other player gets one last chance.

Live: farkle.l8a.dev

Stack

  • Bun — runtime, server, bundler, and test runner; handles everything from WebSocket connections to building the frontend in a single binary
  • WebSockets (via Bun's built-in server) — real-time communication between all players in a room; no polling, no external library
  • React 19 — UI
  • TypeScript — type safety throughout
  • Tailwind CSS 4 — styling

Architecture

In-Memory Game State

There is no database. All room and game state lives in server memory for the duration of a session. This keeps the stack small and the latency low — a turn update is a WebSocket broadcast, not a round-trip to a database. The tradeoff is that a server restart clears all active rooms, which is an acceptable cost for a casual game with sessions measured in minutes. This may be the target of a future update.

WebSocket Communication

Each room is a WebSocket subscription group. When a player takes an action — rolling, selecting dice, banking — the server updates the authoritative room state and broadcasts the new state to every connected client in that room. Clients are purely reactive: they render whatever state the server sends and emit events when the player acts. There is no client-side game logic that diverges from the server.

Shared Scoring Logic

Scoring rules are defined once and used in two places: the server validates every move against them before accepting it, and the client uses the same logic to highlight which dice selections are legal before the player even submits. Keeping both sides in sync this way means the UI can give instant feedback without trusting the client to enforce rules — the server is always the authority.

Session Persistence

Player identity is stored in localStorage so that a page refresh or accidental close does not drop someone from the game. When a client reconnects it presents its stored session ID and the server reattaches it to the correct room and player slot. This avoids the need for accounts or any server-side session store.

Endgame Flow

Reaching 10,000 points does not immediately end the game. The server transitions the room into a final-round state and works through the remaining players in turn order, giving each one last chance to beat the leader's score. Only after all remaining players have had their turn does the server resolve the winner and broadcast the final standings.
