One-Timer

A web application for securely sharing sensitive information—API keys, passwords, confidential messages—through self-destructing one-time links. The sender pastes a secret into the UI, receives a shareable link, and once the recipient opens it the secret is permanently deleted from the server. It also expires automatically after 24 hours if never viewed.

The source is not public, but the concepts behind it are straightforward enough to walk through.

Live: onetimer.l8a.dev

Stack

  • Bun — runtime, server, bundler, and test runner; replaces Node.js, Express, and a separate build tool in one binary
  • SQLite (via Bun's built-in driver) — persistent storage for encrypted secrets, running in WAL mode for write concurrency
  • AES-256-CBC (Node.js Crypto API) — symmetric encryption for secrets at rest
  • React 19 — UI
  • React Router 7 — client-side routing
  • Tailwind CSS 4 — styling
  • TypeScript — type safety throughout
  • PM2 — process management on a VPS

Architecture

Encryption Model

The central design decision is that the server never holds the encryption key. When a secret is submitted, the server generates a random 256-bit key and a random initialization vector, encrypts the secret with AES-256-CBC, stores only the ciphertext and initialization vector, and returns the key to the browser. The browser then constructs the shareable URL with the key embedded in the query string. The server has no way to decrypt what it stored—only someone with the full link can.

One-Time Retrieval

When a recipient opens the link, the browser extracts the secret ID and key from the URL and sends both to the server. The server looks up the ciphertext, decrypts it using the supplied key, deletes the record from the database, and returns the plaintext. The deletion happens in the same operation as the read, so the secret cannot be retrieved a second time regardless of what the client does next.

Expiration

Each secret is stored with an expiration timestamp set 24 hours in the future. Retrieval checks the timestamp before decrypting; if the deadline has passed the record is treated as if it does not exist. This puts a firm upper bound on how long unviewed secrets linger on the server.

Frontend Flow

The app has two views. The home view handles secret creation: the user types a secret, submits the form, and the resulting link is displayed with a copy button. The reveal view handles retrieval: the recipient lands on the URL, sees a prompt to confirm they want to view the secret, and clicks through to trigger the API call. Gating the retrieval behind a deliberate action prevents prefetch bots and link previews from consuming the secret before the intended recipient sees it.

Server & Build

Bun serves both the API routes and the static frontend from a single process. The frontend is bundled at startup using Bun's built-in bundler with a Tailwind plugin, so there is no separate build step — the server handles it.
